# Phase 1 — Trust & correctness (criticals + failing tests)

## P1-1 — Marketplace inquiry must not write plaintext PII — **P0**

### What & Where (verified)
`confirmMarketplaceInterest` (`apps/web/app/actions/marketplace.ts:467-475`) creates a seller-side CRM customer with `phone: payload.contactPhone?.trim() || "—"` — raw plaintext into the encrypted column, no `phoneHash`, `"—"` placeholders. The canonical path (`apps/web/app/actions/customers.ts:139-155`) encrypts via `encryptCustomerData()` (`apps/web/lib/pii-crypto.ts`) and writes the blind-index hashes. Marketplace-created customers are therefore invisible to hash search and break the masking model.

### Why it matters
PDPL/NDMO: plaintext in a column represented as encrypted is exactly the "failure to safeguard" class SDAIA enforcement targets (48 enforcement decisions issued 2025–26 — iclg.com KSA data-protection survey). Record the fix + retroactive repair as an NDMO controls evidence item.

### Implementation plan
1. **Require `contactPhone`** in the inquiry payload (zod, Saudi-mobile validation via the new `normalizeSaudiPhoneE164` — see P1-2 step 3). Reject invalid; **no `"—"` placeholder**. (Alternative — nullable `Customer.phone` — rejected: schema change for no product win; a B2B inquiry needs a callback number.)
2. Route the create through `encryptCustomerData()` exactly like `customers.ts:139-155` (phone + phoneHash; name fields unencrypted as today). Keep it inside the existing `db.$transaction`.
3. **Repair script** `packages/db/scripts/repair-marketplace-customer-pii.ts` (precedent: `encrypt-existing-pii.ts`): select customers `source: "MARKETPLACE"` whose `phone` does not match the 3-part `iv:authTag:ciphertext` format (`apps/web/lib/encryption.ts:19-29`); encrypt + hash real values; `"—"` rows → set a sentinel the UI treats as missing. Idempotent, batched. **Supervised prod run — listed as an ops step in the release checklist.**
4. **Lint guard (mechanism over discipline):** ESLint `no-restricted-syntax` in `packages/eslint-config` banning `*.customer.create|update|upsert` member calls outside `app/actions/customers.ts` + the marketplace helper — same technique as the raw-button rules (§6.6.0).
5. Tests: Vitest on payload validation/normalization (pure); E2E (in the new P4-1 project): cross-org inquiry → seller CRM shows the contact, hash search finds it by phone.

## P1-2 — CRM Kanban accessibility + contact-link gating — **P0** (failing axe E2E)

### What & Where (verified)
- Card container `apps/web/app/dashboard/crm/CrmView.tsx:1609-1623`: `<div draggable role="button" tabIndex={0} onClick>` with nested buttons/links → axe **serious `nested-interactive`** (the failing CRM spec). DnD = native HTML5 (`draggable` + `onDragStart`), no library.
- Contact links: card `:1548-1550` derives `hasPhone` (checks `*` masking but not ciphertext/garbage digits) → `tel:`/`wa.me` at `:1738-1760`; **drawer `:608-643` builds `tel:${customer.phone}` with no validation at all** (masked `******4567` becomes a clickable broken link).
- Data path: `getCustomer()` (`customers.ts:179-202`) → `decryptCustomerData()` → `maskCustomerPii(decrypted, session.can("customers:read_pii"))`.

### Best-practice basis (researched)
- Deque/axe remediation is **structural** — `stopPropagation` does not fix it (the violation is in the accessibility tree/focus order, not event bubbling): dequeuniversity.com/rules/axe/4.8/nested-interactive.
- The accepted fix is the **redundant-click card** (Heydon Pickering, inclusive-components.design/cards): container is a plain `div`; ONE real control inside (card title) is the accessible open-affordance; container click is pointer-only convenience that forwards to it. Elastic EUI fixed the same axe violation this way (EUI PR #6517).
- Atlassian Pragmatic-DnD a11y guidance: every drag outcome must be achievable without dragging, via an action menu — not arrow-key drag modes (atlassian.design/components/pragmatic-drag-and-drop/accessibility-guidelines).
- Masked/missing phone → **omit the control, never disable** (Adrian Roselli, "Don't Disable Form Controls"). `wa.me` format = full international number, digits only, no `+`/zeros (faq.whatsapp.com/5913398998672934).

### Implementation plan
1. **Card structure:** remove `role="button"`, `tabIndex`, card-level `aria-label` from the draggable container (keep `draggable`/`onDragStart`). Card title becomes the single real open-profile control (governed primitive per §6.6.0 — this also re-satisfies the §6.6.7 one-affordance rule). Container `onClick` forwards unless `event.target.closest('a,button')`. Contact `IconButton`s are now siblings in the a11y tree → violation gone.
2. **Accessible move affordance:** card overflow menu gets "Move to <stage>" items (keyboard/SR path for what drag does). Minimal scope: menu items calling the existing stage-change handler; live-region announcement of old → new stage.
3. **New `apps/web/lib/phone.ts`:** `normalizeSaudiPhoneE164(raw: string): string | null` — accepts `05XXXXXXXX`, `+9665XXXXXXXX`, `9665XXXXXXXX`; returns `+9665XXXXXXXX` or `null` (masked `*`, ciphertext `:`-format, junk → null). **One function used at write AND search time** (P1-1, P2-3) — blind-index practice requires identical normalization on both sides (unifiedid.com normalization spec).
4. **Single gating seam:** `getCustomer`/list mapping adds derived `contactPhoneE164: string | null` (computed AFTER decrypt+mask — masked values yield null). Drawer (`:608-643`) and card (`:1548-1550`, `:1738-1760`) render Call/WhatsApp **only when non-null**: `tel:${contactPhoneE164}`, `wa.me/${contactPhoneE164.slice(1)}`.
5. Tests: Vitest table for `normalizeSaudiPhoneE164`; the failing CRM axe spec goes green; manual keyboard pass (Tab reaches title + contact actions as separate stops, Enter opens profile).

### §3.9 note
CRM is the heaviest-touched route this release: 4 theme/lang screenshots + keyboard Tab-through + mobile 375×812 + console-clean, evidence in chat before tag.

## P1-3 — Move `getPlans` out of `"use server"` + lint rule — **P0**

### What & Where (verified)
`apps/web/app/actions/billing.ts:1,21-32`: `"use server"` file exports `const getPlans = unstable_cache(...)` — a non-async-function export. This is the **documented v4.7.0 landmine** (AGENTS.md §4: `tsc` and Playwright don't catch it; the route's action bundle collapses at runtime).

### Best-practice basis (researched)
- Official: `"use server"` files may only export async functions (nextjs.org/docs/messages/invalid-use-server-value); `unstable_cache` export from actions files is a known won't-fix (vercel/next.js#62926).
- Correct home: a **server-only DAL module** — also a security win: an exported `"use server"` loader is an unauthenticated POST endpoint (nextjs.org/docs/app/guides/data-security).
- `unstable_cache` is deprecated-but-working in Next 16; `"use cache"` requires the app-wide `cacheComponents` opt-in → **keep `unstable_cache` now**; keep tag names stable for the future migration (nextjs.org/docs/app/api-reference/functions/unstable_cache).
- **No official lint rule exists** for this class — custom rule is the only pre-build catch.

### Implementation plan
1. New `apps/web/lib/server/plans.ts`: `import "server-only"` at top, NO `"use server"`; move the `unstable_cache` block verbatim (keep key `["public-plans"]` + tag `plans`). Swap its `JSON.parse(JSON.stringify())` for `serialize()` from `apps/web/lib/serialize.ts` while moving (F6 adoption, touched-file rule).
2. Update all `getPlans` importers to the new module; remove the export from `billing.ts`.
3. **Custom ESLint rule** in `packages/eslint-config` (`no-non-async-export-in-use-server`): if `Program.body[0]` is the `"use server"` directive, flag any export whose declaration is not an async function. Run repo-wide; fix any additional hits in the same commit. This mechanizes the AGENTS.md §4 rule — update §4 to reference the rule when it ships.

# Phase 2 — Hardening

## P2-1 — Registration rate limiting — P1

### What & Where (verified)
`registerUser` (`apps/web/app/actions/auth.ts:82-160`) creates an Organization + ADMIN from public input with zero abuse controls. The canonical limiter `checkRateLimit(key, limit, windowMs)` (`apps/web/lib/rate-limit.ts` — atomic Postgres UPSERT, fails open) is proven post-v4.17 (`password.ts:63`, `onboarding.ts:40`).

### Best-practice basis (researched)
OWASP Authentication Cheat Sheet requires anti-automation on registration endpoints; key **both per-IP and per-normalized-email** (per-IP alone over-blocks carrier-grade NAT — common in KSA; per-email alone is defeated by `+aliasing`). Fixed-window is fit for purpose at signup budgets — boundary-burst worst case ≈ 2× a small limit (blog.arcjet.com rate-limiting algorithms).

### Implementation plan
1. In `registerUser`: `headers()` from `next/headers` → first hop of `x-forwarded-for` (fallback: skip the IP check, keep the email check — never throw on a missing header).
2. `checkRateLimit(\`register:ip:${ip}\`, 5, 60*60*1000)` and `checkRateLimit(\`register:email:${normalizedEmail}\`, 3, 60*60*1000)`; email normalization: lowercase + strip `+suffix` before keying.
3. Return `{ error: "RATE_LIMITED" }` matching the existing auth error contract (login already handles RATE_LIMITED strings — `auth.ts:60-62`). Friendly copy per §6.11.4, AR + EN.
4. Verify: hit the limit locally; confirm the registration form surfaces the friendly error in both languages.

## P2-3 — Hash-search hardening: normalization + backfill + indexes — P1 (the one schema change)

### What & Where (verified — audit claim CORRECTED)
- The audit called the hash columns "SHA-256". **Wrong in the dangerous direction only in the schema comment:** `hashForSearch` (`apps/web/lib/encryption.ts:75-77`) is already `"v1:" + HMAC-SHA256(value.trim().toLowerCase(), PII_HASH_PEPPER)`, failing closed. The keyed-hash requirement (CipherSweet/blind-index practice — a plain hash over the ~10⁸ Saudi-mobile keyspace would be reversible in seconds) is **already satisfied**. Fix the misleading schema comment.
- Real gaps: **(a)** no normalization beyond trim/lowercase → `0551234567` vs `+966551234567` hash differently and exact search silently misses; **(b)** hash columns (`schema.prisma:218-220`) have no indexes; **(c)** `@@index([nationalId])` (`:250`) indexes random-IV ciphertext — dead weight.

### Implementation plan
1. `schema.prisma` Customer: add `@@index([organizationId, phoneHash])`, `@@index([organizationId, emailHash])`, `@@index([organizationId, nationalIdHash])` (composite btree — hash-type indexes are single-column/equality-only and can't be tenant-scoped; btree is the documented default). **Remove** `@@index([nationalId])`. Correct the `// SHA-256 hash` comments to `// HMAC-SHA256 blind index (lib/encryption.ts hashForSearch)`.
2. `npx prisma db push` — additive index-only diff, run plain (no `--accept-data-loss`) per AGENTS.md §4. **No new tables → no RLS script change**; CI `rls:check` confirms.
3. `encryptCustomerData` (`lib/pii-crypto.ts`) + the search path (`customers.ts:278-287`): normalize phone via `normalizeSaudiPhoneE164` (P1-2 step 3) before `hashForSearch`. Same function both sides — that is the entire point.
4. **Backfill** `packages/db/scripts/rehash-customer-phones.ts`: decrypt each customer phone → normalize → recompute `phoneHash` where changed. Idempotent, batched, logged counts. **Supervised prod run — ops step in the release checklist** (precedent: `encrypt-existing-pii.ts`).
5. Tests: Vitest — hash(normalize("05…")) === hash(normalize("+9665…")); search E2E finds a customer by either format.
